redis简介

  • redis是一种kv类型的非关系型数据库，数据全部存储在内存当中，所以当重启或者宕机时数据会全部丢失，不过支持数据持久化，将数据保存在磁盘中，在重启的时候加载将恢复，支持集群部署，所以redis在对外提供高性能服务的同时，也保证了自己本身的高可用。
  • redis是单线程（redis6.0之后支持多线程），因为作者认为redis的操作是纯内存操作，所以redis的性能瓶颈不在CPU而是带宽，而使用单线程减少了线程切换带来的资源消耗，不用考虑各种锁的问题，同时使用了多路I/O复用，能够并发处理多个请求。

安装部署

  • 下载安装包：wget http://download.redis.io/releases/redis-5.0.8.tar.gz
  • 创建安装目录：mkdir -p /usr/local/redis
  • 解压：tar zxvf redis-5.0.8.tar.gz -C ./ && mv redis-5.0.8 /usr/local/redis
  • 编译安装：cd /usr/local/redis/redis-5.0.8/ && make && make install
  • 将 REDIS 安装为系统服务并后台启动：sh utils/install_server.sh 全部默认即可，提示Installation successful! 即成功
  • 查看REDIS服务启动情况：systemctl status redis_6379.service
  • 设置允许远程连接：vim /etc/redis/6379.conf 将 bind 127.0.0.1 修改为 0.0.0.0 然后重启即可：systemctl restart redis_6379.service
  • 设置访问密码：vim /etc/redis/6379.conf 找到 #requirepass foobared 取消注释并将foobared 自己想要的密码，保存重启
  • 默认端口是6379，可以根据需求调整，配置文件即/etc/redis/6379.conf

事务

  • 命令

    命令 含义
    Multi 标记一个事务块的开始
    Exec 执行所有事务块内的命令
    Discard 取消事务，放弃执行事务块内的所有命令
    Watch 用于监视一个(或多个) key ，如果在事务执行之前这个(或这些) key 被其他命令所改动，那么事务将被打断
    UNWATCH 取消 WATCH 命令对所有 key 的监视

  • 支持事务的ACID四大特性

  • 事务中的所有命令都会被串行执行

  • 事务开始之后不会再为其他客户端提供服务

  • 队列中的任务要么全部处理，要么全部丢弃

  • 如果开启了AOF，写操作会全部会被写入到aof文件中，防止宕机

  • 不支持回滚，在exec之前，会对添加到队列中的命令进行检查，如果检查通过后，再发生exec之后执行发生错误，该错误命令会被忽略，其他命令继续执行，因为redis认为如果，事务出错，就是你代码写的有问题，和redis无关

    过期策略

  • 惰性删除：每次访问key的时候检查是否过期，如果过期则删除，该操作能节省cpu资源，但是会出现内存的浪费

  • 定时删除：在对key设置过期时间的同时，为该key创建一个定时器，key过期时对key立即进行删除，能够保证内存被尽快释放叼，但是定时器会对cpu带来资源消耗

  • 定期删除：定时任务，每次对设置了过期时间的key(部分)进行扫描，对过期的key，进行删除，是上面两种方案的折中选择

  • RDB持久化

    • 从数据库持久化数据到RDB文件：在持久化之前会检查key是否过期，对过期的key不导入RDB文件中
    • 从RDB文件恢数据到数据库中：数据载入数据库之前会对key进行过期检查，如果过期不导入数据库

  • AOF持久化：

    • 从数据库中持久化到AOF文件：

      当key过期后还没被删除，此时进行持久化操作(该key是不会写入AOF文件中，因为没发生修改命令)

      当key过期后，发生删除操作是，del命令会被写入到aof文件中

    • AOF重写：

      重写是会判断key是否过期，过期是不会写入aof文件中

  • redis使用惰性删除和定期删除，和通过redis.conf中配置hz来配置扫描频率，默认为10，每秒进行10次过期扫描：

    • 从过期字典中随机20个key

    • 删除这20个key中已过期的

    • 如果超过25%的key过期，则重复第一步

    • 设置了扫描的时间上限，默认不会超过25ms

淘汰机制

  • 在/etc/redis/6379.conf可以配置redis使用的最大内存 maxmemory 100mb,或者通过命令设置config set maxmemory 100mb,64位系统默认是0，无限制，32位系统最多不能超过3GB,当redis使用内存达到阈值的时候，就会触发淘汰机制:

    • **noeviction(默认策略)**：对于写请求不再提供服务，直接返回错误（DEL请求和部分特殊请求除外）

    • allkeys-lru：从所有key中使用LRU算法进行淘汰

    • volatile-lru：从设置了过期时间的key中使用LRU算法进行淘汰

    • allkeys-random：从所有key中随机淘汰数据

    • volatile-random：从设置了过期时间的key中随机淘汰

    • volatile-ttl：在设置了过期时间的key中，根据key的过期时间进行淘汰，越早过期的越优先被淘汰

  • 查看当前淘汰策略：config get maxmemory-policy

  • 设置淘汰策略:

    • 修改配置文件：maxmemory-policy allkeys-lru
    • 命令修改：config set maxmemory-policy allkeys-lru

  • LRU

    Redis使用的是近似LRU算法，它跟常规的LRU算法还不太一样。近似LRU算法通过随机采样法淘汰数据，每次随机出5（默认）个key，从里面淘汰掉最近最少使用的key,可以通过maxmemory-samples参数修改采样数量。

    Redis3.0对近似LRU算法进行了一些优化。新算法会维护一个候选池（大小为16），池中的数据根据访问时间进行排序，第一次随机选取的key都会放入池中，随后每次随机选取的key只有在访问时间小于池中最小的时间才会放入池中，直到候选池被放满。当放满后，如果有新的key需要放入，则将池中最后访问时间最大（最近被访问）的移除。

  • LFU

    LFU算法是Redis4.0里面新加的一种淘汰策略:

    • volatile-lfu：在设置了过期时间的key中使用LFU算法淘汰key
    • allkeys-lfu：在所有的key中使用LFU算法淘汰数据

  • 数据结构

    当想要添加一个新元素到整数集合中时，并且新元素的类型比整数集合现有的所有元素的类型都要长，整数集合需要先进行升级（upgrade），才能将新元素添加到整数集合里面。每次想整数集合中添加新元素都有可能会引起升级，每次升级都需要对底层数组已有的所有元素进行类型转换:

ziplist

  • 压缩列表（ziplist）是为了节约内存而设计的，是由一系列特殊编码的连续内存块组成的顺序性（sequential）数据结构，一个压缩列表可以包含多个节点，每个节点可以保存一个字节数组或者一个整数值

  • 压缩列表是列表（List）和散列（Hash）的底层实现之一，一个列表只包含少量列表项，并且每个列表项是小整数值或比较短的字符串，会使用压缩列表作为底层实现（在3.2版本之后是使用quicklist实现）

    • zlbytes：记录整个压缩列表占用的内存字节数，在压缩列表内存重分配，或者计算zlend的位置时使用
    • zltail：记录压缩列表表尾节点距离压缩列表的起始地址有多少字节，通过该偏移量，可以不用遍历整个压缩列表就可以确定表尾节点的地址
    • zllen：记录压缩列表包含的节点数量，但该属性值小于UINT16_MAX（65535）时，该值就是压缩列表的节点数量，否则需要遍历整个压缩列表才能计算出真实的节点数量
    • entryX：压缩列表的节点
    • zlend：特殊值0xFF（十进制255），用于标记压缩列表的末端

  • entryX:压缩包节点构成

    • previous_entry_ength：记录压缩列表前一个字节的长度
    • encoding：节点的encoding保存的是节点的content的内容类型
    • content：content区域用于保存节点的内容，节点内容类型和长度由encoding决定

持久化

RDB

  • RDB是对当前数据库状态的一种快照，将当前数据库中的数据生成二进制的RDB文件，当数据库重启或者宕机的时候使用RDB文件对数据进行恢复

  • RDB既可以手动执行，也可以配置定期执行,有2个命令可以生成RDB文件，SAVE、BGSAVE。

    • SAVE会阻塞redis，直到RDB文件创建完成，在阻塞期间不对外提供任何服务；
    • BGSAVE会生成一个子进程，让后由子进程负责创建RDB文件，父进程继续对外提供服务
    • 有源码可知，save是通过rdbSave触发rdb文件的生成，而bgsave是通过创建子进程执行rdbsave

AOF

  • AOF持久化通过记录执行的写命令，来保存数据库当前的状态
  • AOF会将执行的写命令追加到一个缓存区
  • 更具设置将缓冲区中的命令同步到硬盘（默认是1s同步一次）
  • 执行bgrewriteaof会对aof文件进行重写
  • 当AOF文件损坏是可以使用redis-check-aof fix对文件进行修复

RDB、AOF配置

  • RDB配置

    • 在/etc/redis/6379.conf配置save 60 1000:

      每隔60s，如果有超过1000个key发生了变更，那么就生成一个新的dump.rdb文件save可以配置多个

  • AOF配置

    • 在/etc/redis/6379.conf配置appendonly yes

    • aof刷新时间间隔可用过appendfsync配置：

      • alaways：每一条都写入磁盘，性能影响大
      • everysec：每秒写入磁盘，性能影响小（默认）
      • no：不进行设置，让操作系统自动保存

    • aof rewrite：

      auto-aof-rewrite-percentage 100：aof文件超过原来aof文件容量一倍时，进行aof重写，100为超过原文件大小100%

      auto-aof-rewrite-min-size 64mb： 执行aof重写是文件最小容量

RDB、AOF优缺点

  • RDB优缺点：

  • 优点：

    • RDB是某一时间点的快照，是一个紧凑的单文件，更多用于数据备份。可以按每小时或每日来备份，方便从不同的版本恢复数据。
    • 单文件容易传输到远程服务做故障恢复。
    • RDB可以Fork子进程进行持久化，使Redis可以更好地处理用户请求
    • 在大量数据的情况下，RDB相比较于AOF会更快的加载。

  • 缺点：

    • 如果Redis不及时保存RDB文件，会造成数据的丢失。例如系统突然断电，但未来得及保存数据。即使你设置更多的Save point，也无法保证100%的数据不丢失。
    • RDB经常需要fork子进程去执行，但如果再大量数据的情况下，这个fork操作会非常耗CPU资源的。对比AOF虽然也是fork，但是它的数据保存处理是可以控制的，不需要全量保存。

  • AOF优缺点

    • 优点：

      • AOF可以设置 完全不同步、每秒同步、每次操作同，默认是每秒同步。因为AOF是操作指令的追加，所以可以频繁的大量的同步。
      • AOF文件是一个值追加日志的文件，即使服务宕机为写入完整的命令，也可以通过redis-check-aof工具修复这些问题。
      • 如果AOF文件过大，Redis会在后台自动地重写AOF文件。重写后会使AOF文件压缩到最小所需的指令集。
      • AOF文件是有序保存数据库的所有写入操作，易读，易分析。即使如果不小心误操作数据库，也很容易找出错误指令，恢复到某个数据节点。例如不小心FLUSHALL，可以非常容易恢复到执行命令之前。

    • 缺点

      • 相同数据量下，AOF的文件通常体积会比RDB大。因为AOF是存指令的，而RDB是所有指令的结果快照。但AOF在日志重写后会压缩一些空间。
      • 在大量写入和载入的时候，AOF的效率会比RDB低。因为大量写入，AOF会执行更多的保存命令，载入的时候也需要大量的重执行命令来得到最后的结果。RDB对此更有优势。

  • 复制超时

    如果数据比较多，rdb文件比较大的话生成rdb文件到将rdb文件传输给slave，可能会超时，从节点拒绝接受rdb文件，默认60s，可以调整repl-timeout参数

集群模式

主从复制模式

通过持久化功能，Redis保证了即使在服务器重启或者宕机的情况下也不会有数据丢失（或者少量丢失），但是如果数据指存储在一台服务器上当这台服务器的硬盘出现了故障，其他服务器依然可以提供服务。

sentinel（哨兵模式）

主从模式当主节点宕机后需要手动把一个从节点切换切换为主节点，会造成一段时间服务的不可用

哨兵模式是一种特殊的模式，哨兵是一个独立的进程，会实时向每个节点发送命令监控其状态，当哨兵发现master宕机后会自动将slave切换成master，并且让其他的从节点修改配置文件。

当时当主节点处于正常状态时，哨兵与主节点断开连接，那么主节点就被误判了，哨兵会选取一个从节点作为主节点，这时候存在2个主节点，这种现象被称为集群脑裂，所以哨兵也需要进行监控，多个哨兵与互相监控，当其中一个哨兵发现主节点不可用，此时被称为主观下线，当其他的哨兵也检测到主节点不可用，并达到一定数量时，才认为主节点时真正的不可用，此时称为客观下线。

集群模式

哨兵模式每个节点都会存储相同的数据，在一定情况下时对资源的极大的浪费，所以集群模式时每个节点上存储了不通的数据，集群模式任意2个节点都是相通的，但是为了保证集群模式的高可用，会给每个节点配1个或多个从节点，redis集群分片使用的时hash槽，16384 个hash槽

常见问题

缓存击穿

查询的数据在数据库中是存在，但是缓存中热点数据key恰巧失效，所有请求直接怼到数据库导致数据库连接等待甚至宕机

使用

命令 含义
PFADD KEY VALUE1 VALUE2…VALUEN 添加元素
PFCOUNT KEY 获取元素个数（去重后）
PFMERGE NEWKEY KEY1 KEY2…KEYN 合并多个key

原理

HyperLogLog 用于不精确的基数统计HyperLogLog 提供了两个指令 pfadd 和 pfcount，根据字面意义很好理解，一个是增加计数，一个是获取计数。 pfadd 用法和 set 集合的 sadd 是一样的，来一个用户 ID，就将用户 ID 塞进去就是。 pfcount 和 scard 用法是一样的，直接获取计数值。

  • 伯努利试验

    • 抛硬币出现正反面的概率都是1/2，一直抛硬币直到出现正面，记录下投掷次数k，将这种抛硬币多次直到出现正面的过程记为一次伯努利过程，对于n次伯努利过程,我们会得到n个出现正面的投掷次数值k1, k2, k3……kn，其中最大值记为k_max，可得出如下结论：

      • n 次伯努利过程的投掷次数都不大于 k_max
      • n 次伯努利过程，至少有一次投掷次数等于 k_max

      最终根据极大估算方法可得出n=2^(k_max)

    • 对上述实验做m轮，取每轮的k_max 相加的k_mx取平均值可得k_average=k_mx/m，在进行估算：

      N=constant*m*2^k_average constant是修正因子不用管他怎么来的，我也不知道。也别管公式怎么推到出来的，也不知道。
